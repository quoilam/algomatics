## 当前系统的本质

### 多Agent工作流编排系统 (Workflow Orchestration)

当前系统采用**固定流程**设计：

```
搜索 (Retrieval) → 生成 (CodeGen) → 执行 (Execution) → 评估 (Evaluation)
```

#### 系统特征

**架构**：
- 每个Agent职责单一且清晰
- 数据在Agent间通过Controller显式传递
- 流程顺序预定义，不依赖运行时状态

**优势**：
- ✅ **可维护性强**：每个Agent代码独立，易于理解和测试
- ✅ **可扩展性好**：添加新Agent或修改现有Agent不影响其他部分
- ✅ **易于测试**：单个Agent可独立单元测试，流程可集成测试
- ✅ **问题定位容易**：某个步骤失败时，能精确定位问题所在
- ✅ **结果可预测**：同样的输入总是走同样的流程

**限制**：
- ❌ **缺乏灵活性**：无法跳过不必要的步骤或调整执行顺序
- ❌ **无自适应能力**：不能根据中间结果调整策略
- ❌ **反馈不驱动迭代**：评估结果只用于展示，不影响后续执行
- ❌ **无学习能力**：每次都从零开始，无法从历史经验学习
- ❌ **缺乏决策能力**：Controller是编排者而非决策者

## 目标系统的本质

### Agentic自主智能体系统 (Autonomous Agent System)

目标系统的核心特征：**自主性、适应性、反思性、持续迭代**

#### 系统特征

**架构**：
- Controller不仅编排，更进行任务理解和策略规划
- Agent间不仅传递数据，更进行协商和双向反馈
- 执行流程动态生成，依赖任务特征和运行时结果
- 存在反馈循环和知识积累机制

**核心能力**：

1. **自主规划能力**
   - 系统理解用户意图后，自己决定需要哪些步骤
   - 示例：简单问题 → 跳过搜索，直接生成；复杂问题 → 先搜索再生成
   - 决策基于：任务类型、用户约束、系统容量等

2. **闭环反馈能力**
   - 评估结果自动驱动后续迭代
   - 示例：评分<7分 → 自动优化；评分≥7分 → 接受结果
   - 形成内循环：评估→决策→优化→重新评估

3. **自我修正能力**
   - 执行失败时自动分析和修复，无需用户干预
   - 示例：导入错误 → 自动修复→重新执行；逻辑错误 → 分析原因→改进代码
   - 限制条件：防止无限重试，设定修复上限

4. **知识积累能力**
   - 从成功案例中提取模式，指导后续任务
   - 示例：遇到"降噪"问题 → 推荐已验证的算法组合；识别失败模式 → 避免重复错误
   - 学习机制：历史记录→模式识别→推荐→用户反馈→更新知识

## 二者的关键差异

### 维度对比表

| 维度 | 当前系统 | 目标系统 |
|------|---------|---------|
| **设计理念** | 编程式 (Programmatic) | 声明式 (Declarative) |
| **决策方式** | 预定义流程 | 运行时规划 |
| **反馈角色** | 信息展示 | 驱动迭代 |
| **错误处理** | 一次性执行 | 自动诊断和修复 |
| **Agent关系** | 单向数据流 | 双向协商 |
| **学习机制** | 无 | 持续累积 |
| **适应性** | 低 | 高 |
| **用户干预** | 高（需手动反馈） | 低（自动处理） |

## 为什么要转变

### 业务驱动

1. **用户体验**
   - 当前：用户需要多次反馈和调整才能得到满意结果
   - 目标：系统自动优化，用户一次请求即得满意结果

2. **效率提升**
   - 当前：每个用户都要走完整的搜索→生成→执行→评估
   - 目标：简单问题跳过非必要步骤；常见问题推荐已验证方案

3. **成本降低**
   - 当前：每次都调用搜索API，重复计算
   - 目标：知识库复用、缓存优化、策略化执行

### 技术驱动

1. **LLM能力进步**
   - 现在的LLM足以理解任务意图和制定计划
   - 支持更复杂的推理和决策任务

2. **系统成熟度提升**
   - 当前Agent已经稳定可靠，可以加上决策层
   - 有充分的观察数据用于模式提取

3. **架构演进必需**
   - 工作流系统 → 智能系统的自然进化路径
   - 行业趋势：从自动化向自治化发展

## 转变的必要性

### 当前系统的天花板

**无法解决的问题**：
- 用户要求："这个地方不好，帮我改进" → 系统无法自动改进
- 用户要求："帮我快点处理这个问题" → 系统无法跳过不必要步骤
- 系统观察："这类问题经常这样处理成功" → 系统无法利用这个观察
- 系统遇到："这个错误很常见，我能自己修" → 系统只能报错等用户

### Agentic系统的价值

**能够解决的问题**：
- 📊 **智能化决策**：根据任务类型智能选择执行策略
- 🔄 **自动迭代**：评估不满意时自动优化而非等待用户反馈
- 🛠️ **自我修复**：常见错误可自动修复，用户无感
- 💡 **知识复用**：历史经验指导当前决策
- ⚡ **性能优化**：根据条件跳过不必要步骤，加快响应

## 转变的约束和风险

### 什么不变

- ✅ Agent框架保持 - 不重构现有的四大Agent
- ✅ 数据模型保持 - 不改变Agent间的数据格式
- ✅ API接口保持 - 对前端的API保持向后兼容
- ✅ 功能范围 - 不超出图像处理这个领域

### 什么会变

- 🔄 Controller从编排者升级为决策者
- 🔄 Agent能力增强（识别、诊断、协商）
- 🔄 系统架构引入新的中间层（规划引擎、迭代控制等）
- 🔄 引入知识库和学习机制

### 主要风险

- ⚠️ 复杂度增加 → 需要充分的测试和文档
- ⚠️ 决策可能失误 → 需要反馈机制和降级策略
- ⚠️ 迭代陷入循环 → 需要安全机制和终止条件
- ⚠️ 系统变得"不可预测" → 需要解释和可观测性

详见 [05-风险识别与应对](05-risks-mitigation.md)

## 总结

### 本质区别

```
当前系统 = 多个高质量Agent + 预定义流程
           (好的零件 + 刚性流程)

目标系统 = 多个高质量Agent + 智能决策中心 + 知识库
           (好的零件 + 灵活大脑 + 记忆系统)
```

### 进化方向

```
工作流编排系统 (当前)
         ↓
    引入决策能力
         ↓
    引入反馈驱动
         ↓
    引入自我修复
         ↓
    引入知识学习
         ↓
Agentic自治系统 (目标)
```
