# 03 分阶段迭代策略

**📍 位置**：[← 上一篇](02-dimension-comparison.md) | [返回导航](design.md) | [下一篇 →](04-architecture-design.md)

---

## 概览

将Agentic系统的建设分为5个阶段，从最小化可行产品（MVP）开始，逐步增强系统能力。

### 各阶段时间和优先级

| 阶段 | 目标 | 周期 | 优先级 | 前置条件 |
|------|------|------|--------|---------|
| 1 | 自动迭代能力 | 2-3周 | P0 | 无 |
| 2 | 错误自修复能力 | 2-3周 | P1 | 阶段1 |
| 3 | 主动规划与决策 | 4-6周 | P0 | 阶段1 |
| 4 | Agent协作机制 | 4-5周 | P1 | 阶段1+3 |
| 5 | 知识库与学习 | 6-8周 | P2 | 阶段1+3 |

**总计**：21-31周（约5-8个月）

---

## 阶段1：自动迭代能力 🎯 MVP（2-3周）

### 目标

实现最小化的Agentic特征 - **闭环反馈**

这是整个系统升级中最核心的一步。实现此阶段后，系统将具备自动优化的能力，无需等待用户反馈。

### 技术方向

#### 1. 为EvaluationAgent设计结构化评估输出
- 不仅返回文本评价，更返回**量化评分**（0-10）
- 返回**改进建议**（具体的优化方向）
- 返回**关键问题**（如果有的话）

**输出格式示例**：
```json
{
  "overall_score": 7.5,
  "dimensions": {
    "quality": 8,
    "matching": 7,
    "efficiency": 7
  },
  "improvement_suggestions": [
    "考虑使用更高效的算法",
    "这种边界处理方式有改进空间"
  ],
  "key_issues": [
    "内存使用可能偏高"
  ]
}
```

#### 2. 在Controller中实现评分驱动的决策逻辑
```
评分 ≥ 7 → 接受结果，流程完成
5 ≤ 评分 < 7 → 自动优化（最多重试2-3次）
评分 < 5 → 标记为需人工审查或重新开始（不要无限重试）
```

#### 3. 定义迭代上下文传递
当进行第N次迭代时，传递给CodeGenAgent：
- 之前的评分和评估报告
- 改进建议
- 之前尝试过的方案（避免重复）

**传递内容示例**：
```
"第一次迭代得分6.5分，评估意见：应使用更高效的算法。
请在之前的代码基础上改进，注重算法效率。"
```

### 实现步骤

1. **修改EvaluationAgent**
   - 增加评分输出模块
   - 设计改进建议生成逻辑
   - 返回结构化JSON而不仅仅文本

2. **修改Controller**
   - 添加"迭代决策引擎"
   - 实现评分判断逻辑
   - 管理迭代计数和状态

3. **修改CodeGenAgent**
   - 支持接收迭代上下文
   - 在迭代时参考前一次的评估反馈

4. **测试和验证**
   - 单元测试：EvaluationAgent的评分逻辑
   - 集成测试：自动迭代的完整流程
   - 端到端测试：多轮迭代是否收敛

### 验收标准

- ✅ EvaluationAgent能输出0-10的量化评分
- ✅ Controller能根据评分自动决定是否继续迭代
- ✅ 迭代次数可控（3-5次以内），不出现无限循环
- ✅ 多轮迭代后的结果质量有显著提升（评分逐次增加）
- ✅ 无需用户反馈，系统自动完成（>80%成功率）

### 预期效果

- 🎉 70%+的请求无需用户反馈即可得到满意结果
- 🎉 用户体验从"需要多次调整"升级到"一次请求搞定"
- 🎉 为后续阶段打下基础（反馈驱动的决策）

## 阶段2：错误自修复能力（2-3周）

### 目标

代码执行失败时自动修复，而不是直接报错给用户。

### 技术方向

#### 1. 增强ExecutionAgent的错误捕获和分类
- 记录完整错误信息、栈追踪、执行上下文
- **错误类型识别**：
  - **环境错误**（缺失库、版本不兼容）
  - **代码错误**（语法错误、逻辑错误）
  - **资源错误**（内存溢出、超时）
- 为每种错误类型编码可修复的建议

#### 2. CodeGenAgent支持**错误修复模式**
- 接收错误信息作为补充上下文
- 不是完全重新生成代码，而是**有针对性地修复**
- 修复提示应该包含：
  - 原始代码的关键部分
  - 具体的错误消息
  - 执行环境信息

**修复请求示例**：
```
"代码执行时出现导入错误：
ModuleNotFoundError: no module named 'nonexistent_lib'

请修复这个错误。可选方案：
1. 使用标准库替代
2. 简化算法，减少依赖

原始代码片段：
import nonexistent_lib
...
"
```

#### 3. Controller实现修复重试循环
```
代码执行 
    ↓
失败？
├─→ 否 → 返回结果
└─→ 是 → 错误分析
        ↓
        可修复？
        ├─→ 否 → 标记失败，返回给用户
        └─→ 是 → 请求CodeGen修复
                ↓
                重新执行（最多3次）
                ↓
                仍失败？
                ├─→ 否 → 成功！
                └─→ 是 → 上报用户
```

### 关键设计决策

#### 修复 vs 迭代的策略分化
- **修复优先级更高**：代码不执行 → 先修复 → 再评估
- **修复次数单独计数**：不占用功能优化的迭代预算
- **修复失败的早期暴露**：不应该自动降级或忽略，而是明确告知用户

#### 定义可修复性边界
- ✅ 可自动修复：导入错误、简单语法错误、参数错误
- ❌ 不易自动修复：算法逻辑错、性能问题、边界情况

### 实现步骤

1. **增强ExecutionAgent**
   - 设计错误类型识别系统
   - 提取可修复的错误特征
   - 生成修复建议

2. **扩展CodeGenAgent**
   - 添加"修复模式"
   - 实现有针对性的代码修复逻辑

3. **修改Controller的异常处理**
   - 实现修复重试流程
   - 管理修复计数
   - 决定何时放弃修复

4. **测试和验证**
   - 测试常见的可修复错误
   - 验证修复成功率
   - 确保修复不会陷入循环

### 验收标准

- ✅ 常见导入错误、简单语法错误可被自动修复
- ✅ 修复成功率 > 70%（针对可修复的错误）
- ✅ 修复失败时的错误信息对用户有帮助（指出具体问题和建议方案）
- ✅ 修复循环不会无限进行（最多3次）

### 预期效果

- 🎉 20-30%的错误可自动修复，用户无感
- 🎉 减少用户的"碰壁感"
- 🎉 系统更加"健壮"和可靠

## 阶段3：主动规划与决策（4-6周）

### 目标

**Controller从编排者升级为决策者**

根据任务特征动态决定执行策略和Agent序列，而不是固定的搜索→生成→执行→评估。

### 技术方向

#### 1. 设计任务意图理解层
- 将用户输入映射到系统能理解的任务类型
- 识别关键约束条件

**任务类型示例**：
```
简单处理：裁剪、缩放、旋转
中等处理：降噪、锐化、增强
复杂处理：风格转换、超分、去水印
其他：多步骤、组合操作
```

**约束条件示例**：
```
性能要求：实时/普通/不限
质量要求：快速/标准/最高
时间限制：秒级/分钟级/无限
```

#### 2. 设计执行策略库
定义：(任务类型 + 约束条件) → (执行策略 + Agent序列)

**策略示例**：
```
策略A（快速处理）：
  任务：简单处理
  约束：性能优先、实时
  执行：CodeGen + Execution（跳过搜索和详细评估）

策略B（标准处理）：
  任务：中等处理
  约束：普通约束
  执行：完整流程（搜索→生成→执行→评估）

策略C（高质量处理）：
  任务：复杂处理
  约束：质量优先
  执行：搜索→生成→执行→详细评估→可能迭代

策略D（保守处理）：
  任务：任何任务
  约束：不确定、需要探索
  执行：先搜索获取context，再全流程
```

#### 3. 实现动态流程编排
根据策略，动态选择：
- 哪些Agent执行
- 执行顺序
- 是否跳过某些步骤
- 是否启用迭代

### 关键设计决策

#### 任务分类的粒度
- 过细导致规则爆炸，难以维护
- 过粗无法体现差异
- **建议**：初期5-10个类别，后期可根据反馈调整

#### 规划策略的实现方式
选项1：配置文件（YAML/JSON）
- 优点：易于修改，无需重新编译
- 缺点：复杂规则难以表达

选项2：DSL（领域特定语言）
- 优点：表达力强，灵活
- 缺点：学习曲线，维护成本

选项3：代码实现
- 优点：最灵活，易于版本控制
- 缺点：需要代码审查才能改动

**建议**：初期用代码实现，后期如果规则变多再考虑抽象

#### 黑盒 vs 透明
用户是否能看到系统做了什么决策和为什么？

**建议**：提供"解释模式"
```
{
  "decision": "采用策略B（标准处理）",
  "reason": "任务类型：降噪处理（中等复杂度）",
  "agent_sequence": ["RetrievalAgent", "CodeGenAgent", "ExecutionAgent", "EvaluationAgent"],
  "iterations_enabled": true,
  "max_iterations": 3
}
```

### 实现步骤

1. **构建TaskParser（任务解析器）**
   - 从用户输入提取任务类型
   - 识别约束条件
   - 输出结构化任务表示

2. **构建PlanningEngine（规划引擎）**
   - 存储策略规则
   - 根据任务选择最优策略
   - 生成执行计划

3. **修改Controller**
   - 在处理请求前先进行规划
   - 根据计划选择性调用Agent
   - 保持向后兼容

4. **测试和验证**
   - 测试任务识别的准确率
   - 验证不同任务的执行策略
   - 对比不同策略的效果

### 验收标准

- ✅ 正确识别 5+ 种不同的任务类型
- ✅ 简单任务执行时间比当前降低 30%+（通过跳过不必要步骤）
- ✅ 复杂任务的执行效果不降低（与当前流程相当或更好）
- ✅ 用户反馈：感受到系统更"聪明"和高效

### 预期效果

- 🎉 简单请求快速响应（跳过不必要搜索）
- 🎉 API调用成本降低（减少搜索调用）
- 🎉 系统表现出"理解"和"决策"能力

## 阶段4：Agent协作机制（4-5周）

### 目标

**Agent不仅传递数据，更进行协商**

建立Agent间的多向通信，提升整体决策和执行质量。

### 技术方向

#### 1. 设计协作消息协议
定义Agent间通信的标准格式和语义。

**协议示例**：
```json
{
  "speaker": "RetrievalAgent",
  "listener": "CodeGenAgent",
  "message_type": "quality_report",
  "payload": {
    "quality_score": 0.6,
    "confidence": 0.7,
    "suggestions": ["搜索结果较差，建议采用保守生成策略"],
    "raw_data_summary": "找到15个结果，大多无关"
  }
}
```

#### 2. 实现关键协作点
选择最高收益的协作点优先实现：

##### 协作点1：CodeGen与Evaluation的前置反馈
```
CodeGenAgent 完成代码 
    ↓
调用"快速评估"（轻量级，仅检查代码结构）
    ↓
Evaluation 快速反馈："代码有问题，建议这样改"
    ↓
CodeGen 据此修改代码
    ↓
再执行完整流程
```
**收益**：避免执行明显有问题的代码，节省时间

##### 协作点2：Execution与Evaluation的双向反馈
```
Execution 执行失败
    ↓
Evaluation 分析错误原因
    ↓
反馈给CodeGen："问题是这个，建议改成那样"
    ↓
CodeGen 有针对性地修复（而不是盲目重新生成）
```
**收益**：提高修复成功率，减少无谓尝试

##### 协作点3：Retrieval质量与CodeGen的动态调整
```
Retrieval 评估搜索结果质量
    ↓
质量低？告诉CodeGen："信心度低，不要过度依赖搜索结果"
    ↓
CodeGen 采用更保守的生成策略（减少外界依赖）
```
**收益**：在搜索结果差的情况下仍能生成可靠代码

#### 3. 实现轻量级评估机制
为了协作点1，需要快速但粗粒度的评估：
```
快速评估（<1秒）：
  - 检查代码语法
  - 检查必需的导入
  - 检查基本逻辑结构
  - 给出初步评分
  
vs

完整评估（执行完成后）：
  - 实际运行代码
  - 检查输出质量
  - 给出详细分析
  - 给出改进建议
```

### 关键设计决策

#### 平衡协作收益与延迟
- 协作应该增加整体时间 <10%
- 但质量提升 >15%
- 需要精心设计协作点的选择

#### 避免协作死锁
- 防止A等B，B等A的情况
- 定义超时和降级策略：如果协作失败，使用默认行为

#### 协作的可禁用性
- 用户或配置可以控制是否启用某些协作
- 便于测试和故障排查

### 实现步骤

1. **设计协作消息协议**
   - 定义消息结构、类型、字段
   - 确保兼容性和扩展性

2. **在Controller中实现协作协调**
   - 协调Agent的协议协商
   - 管理协作关系和权限
   - 处理协作失败的降级

3. **增强Agent的协作能力**
   - 每个Agent学会发送和接收协作消息
   - 根据协作反馈调整行为

4. **测试和验证**
   - 测试协作消息的传递
   - 验证基于协作反馈的行为改变
   - 测试降级策略

### 验收标准

- ✅ Agent能够主动获取需要的上下文，而非被动等待
- ✅ 协作引入的延迟 < 10%，但质量提升 > 15%
- ✅ 系统日志能清晰记录Agent间的协商过程
- ✅ 协作失败时有合理的降级行为

### 预期效果

- 🎉 整个系统表现得更像一个"团队"而不是"流水线"
- 🎉 质量提升同时性能不明显下降
- 🎉 系统的适应性和容错能力更强

## 阶段5：知识库与学习（6-8周）

### 目标

系统从历史案例中学习和推荐，形成持续改进的机制。

### 技术方向

#### 1. 建立问题模式库
```
问题库结构：
  (问题描述, 问题特征) → 问题ID
  
特征提取：
  - 主题关键词：降噪、增强、检测等
  - 图像特征：大小、格式、质量等
  - 用户约束：时间、质量、性能等
```

#### 2. 建立最佳实践库
```
最佳实践记录：
  {
    "problem_id": "denoise_001",
    "solution": "gaussian_blur + median_filter",
    "parameters": {
      "kernel_size": 5,
      "median_strength": 3
    },
    "evaluation": {
      "score": 8.5,
      "times_used": 42,
      "success_rate": 0.95
    }
  }
```

#### 3. 实现相似问题推荐
```
新问题来临 
    ↓
特征提取和匹配
    ↓
查询知识库中的相似问题
    ↓
推荐排序（基于相似度 + 评分 + 使用频率）
    ↓
展示给用户或直接应用
```

#### 4. 实现反馈学习
```
用户认可结果 
    ↓
记录成功案例
    ↓
更新该方案的评分和优先级

vs

用户拒绝结果 
    ↓
记录失败原因
    ↓
降低该方案的优先级或标记为不推荐
```

### 知识库更新策略

| 操作 | 条件 | 动作 |
|------|------|------|
| 新增 | 新问题被解决且用户认可 | 添加到库，评分初始值5 |
| 更新 | 同一方案被多次成功使用 | 提升评分和优先级 |
| 冷却 | 多次失败的模式 | 降低优先级或隐藏 |
| 遗忘 | 长期未被使用 | 自动降低权重 |

### 关键设计决策

#### 知识库的初期简化设计
- 不过度工程化
- 从简单的JSON/SQLite开始
- 暂不考虑向量数据库或深度学习

#### 更新策略的平衡
- 新知识应该快速进入库（帮助后续用户）
- 但也要谨慎（避免错误知识污染库）
- **建议**：新方案首次使用需要至少1-2次成功才能推荐

#### 检索精度 vs 召回率
- 过度严格的匹配可能遗漏相似问题
- 过度宽松的匹配可能推荐不相关的方案
- **建议**：初期采用简单的关键词匹配，观察效果后再优化

### 实现步骤

1. **设计知识库数据结构和存储**
   - 定义问题、方案、评估的数据模型
   - 选择存储介质（JSON、SQLite等）

2. **构建问题识别和分类系统**
   - 从用户输入提取特征
   - 映射到已知问题类型
   - 识别新问题

3. **实现相似问题检索**
   - 建立特征索引
   - 实现相似度计算
   - 实现排序和推荐

4. **构建反馈学习机制**
   - 记录用户反馈
   - 更新知识库
   - 管理知识库版本

5. **测试和验证**
   - 检验推荐的准确率
   - 测试知识库的增长和更新
   - 评估对系统性能的影响

### 验收标准

- ✅ 累积 50+ 成功案例后，重复问题的推荐准确率 > 80%
- ✅ 使用推荐方案的用户成功率比完全从零开始高 30%+
- ✅ 知识库体积控制在合理范围（不超过100MB）
- ✅ 错误知识定期清理或冷却，不污染库

### 预期效果

- 🎉 热点问题快速响应（直接推荐已验证方案）
- 🎉 新用户受益于前人的成功经验
- 🎉 系统展现出"学习"和"改进"的能力
- 🎉 成本继续下降（减少搜索和重复计算）

---

## 各阶段之间的依赖关系

```
阶段1（自动迭代）
    ↓
    ├→ 阶段2（错误修复）
    ├→ 阶段3（决策规划）
    │   ├→ 阶段4（Agent协作）
    │   └→ 阶段5（知识库）
    └→ 可并行推进
```

**建议顺序**：
1. 先完成阶段1，验证自动迭代能力
2. 可并行进行阶段2和3（相对独立）
3. 完成阶段1和3后再进行阶段4
4. 最后进行阶段5（积累足够多的案例后）

| 阶段 | 工作量 | 实际周期 | 总耗时 |
|------|--------|----------|--------|
| 1 | 中等 | 2-3周 | 2-3周 |
| 2 | 中等 | 2-3周 | 4-6周 |
