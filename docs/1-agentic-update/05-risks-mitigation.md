# 05 风险识别与应对

**📍 位置**：[← 上一篇](04-architecture-design.md) | [返回导航](design.md) | [下一篇 →](06-success-metrics.md)

---

这一章识别系统升级中的主要风险，并提供应对策略。

---

## 风险1：系统复杂度爆炸

### 表现

- 引入更多决策逻辑后，系统变得难以理解和维护
- 新增的中间层（TaskParser、PlanningEngine等）增加了交互复杂性
- 多层决策导致问题排查困难（哪里出错了？）
- 代码改动影响范围大，副作用多

### 具体场景

```
问题：用户请求处理失败了
疑点：是TaskParser识别错了？
     还是PlanningEngine选错策略？
     还是Agent执行有问题？
     还是协作消息丢失了？
     
排查时间：可能需要看6个地方的日志
```

### 风险等级

🔴 **高** - 影响系统的可维护性和团队生产力

### 应对策略

#### 1. 采用渐进式迭代
- **不要一次性做完所有5个阶段**
- 完成一个阶段，充分测试和稳定后再进入下一阶段
- 阶段间保持清晰的边界和接口

#### 2. 保持现有Agent的相对独立性
- 最小化对现有四大Agent的改动
- 新能力通过Controller层面的增强来实现
- 保证现有Agent仍能独立运行和测试

#### 3. 充分的单元测试和集成测试
- 每个新组件（TaskParser、PlanningEngine等）都有单元测试
- 组件间的集成有集成测试
- 完整流程有端到端测试

#### 4. 清晰的日志和追踪
```
每个关键步骤都记录日志，包含：
- 输入数据
- 处理结果
- 耗时
- 相关参数

示例：
[TASK_PARSE] input="降噪处理", result={"type": "medium", "score": 0.92}
[PLANNING] task=medium, strategy="standard", agents=[...]
```

#### 5. 架构文档和设计决策记录
- 每个新组件都有清晰的职责说明
- 关键决策都有"为什么"的解释
- 定期更新架构图和流程图

### 预防检查清单

- [ ] 新代码是否有对应的测试？
- [ ] 是否更新了相关文档？
- [ ] 改动是否影响了现有功能？
- [ ] 日志是否足够追踪问题？
- [ ] 代码审查中是否有架构师评审？

## 风险2：决策错误导致体验下降

### 表现

- 规划引擎做出错误的执行策略选择
- 导致比当前更慢或质量更差的结果
- 用户反馈："为什么系统变得更慢了？"
- 信任度下降

### 具体场景

```
场景A：简单任务被识别为复杂任务
  用户：裁剪图片（很简单）
  系统识别：复杂处理（错了！）
  系统决策：启用搜索+完整流程
  结果：不必要的耗时
  
场景B：需要搜索的任务被判定为不需要搜索
  用户：去除水印（需要最新算法）
  系统识别：简单处理（错了！）
  系统决策：跳过搜索，直接生成
  结果：生成效果差，用户失望
```

### 风险等级

🟠 **中高** - 影响用户体验和系统口碑

### 应对策略

#### 1. 初期采用保守策略
- **不要急于优化**
- 初期总是跑完整流程（搜索→生成→执行→评估）
- 只有在充分验证后才跳过某个步骤

#### 2. 制定规则的严谨流程
```
规则提案 → 讨论确认 → 小范围测试 → 
  ↓
  评估效果（对比：有规则 vs 无规则）
  ↓
  效果好（时间快30%+，质量不降）→ 推广
  
  效果差（时间无改善或质量下降）→ 修改或删除
```

#### 3. 提供"解释模式"
让用户理解系统为什么这样做：
```json
{
  "strategy": "fast",
  "explanation": "这是一个简单的图片裁剪任务，我采用了快速策略。",
  "agent_sequence": ["CodeGen", "Execution"],
  "estimated_time": "5秒",
  "why_skip_search": "这类任务已有成熟的OpenCV方案，不需要搜索"
}
```

#### 4. A/B测试框架
- 对部分用户启用新规则，对部分用户保持原有流程
- 对比两组用户的满意度、执行时间、成本
- 以数据驱动的方式评估规则是否可行

#### 5. 快速的降级和回滚机制
- 如果发现规则导致问题，快速关闭该规则
- 回到安全的保守策略
- 不让用户长期受影响

### 预防检查清单

- [ ] 新规则是否经过小范围测试？
- [ ] 是否有对比数据（新规则 vs 原流程）？
- [ ] 用户是否能理解系统的决策？
- [ ] 是否有快速回滚的机制？
- [ ] 监控告警是否能及时发现异常？

## 风险3：迭代陷入死循环

### 表现

- 系统反复优化代码但评分不变
- 在两个状态间震荡（评分在6-7分间浮动）
- 占用计算资源但无法收敛
- 用户等待时间过长

### 具体场景

```
第1次迭代：评分 6.5 → "优化效率"
第2次迭代：评分 6.8 → "优化效率"
第3次迭代：评分 6.4 → "优化速度"（换方向）
第4次迭代：评分 6.7 → "优化效率"（回到原方向）
...
无尽循环，消耗资源
```

### 风险等级

🟠 **中高** - 影响系统效率和用户体验

### 应对策略

#### 1. 严格的迭代上限
- 最多迭代 3-5 次
- 无论是否达到满意评分，超出次数就停止
- 建议：修复错误最多3次，功能优化最多3次

#### 2. 检测评分变化趋势
```python
def should_continue_iteration(scores: List[float]) -> bool:
    """
    评分列表：[6.5, 6.8, 6.4, 6.7, ...]
    趋势判断：
    - 如果连续N次评分无明显进展 → 停止
    - 如果评分在某个范围震荡 → 停止
    - 如果评分下降 → 可能要停止
    """
    if len(scores) < 2:
        return True
    
    # 方案1：检测平均趋势
    last_avg = average(scores[-3:])
    prev_avg = average(scores[-6:-3])
    if abs(last_avg - prev_avg) < 0.2:
        return False  # 无进展，停止
    
    # 方案2：检测最大改进
    improvement = max(scores[1:]) - scores[0]
    if improvement < 0.5 and len(scores) > 3:
        return False  # 改进不足，停止
    
    return True
```

#### 3. 引入随机扰动
在迭代策略上引入一些随机性，避免陷入局部最优：
```
如果连续两次迭代评分无改进：
  ✓ 概率：50% 尝试完全不同的方案
  ✓ 概率：50% 停止迭代
```

#### 4. 设置"突破点"
定义每个迭代需要达成的目标，否则停止：
```
迭代1：评分必须 > 5
迭代2：评分必须 > 6
迭代3：评分必须 > 6.5
迭代4+：评分必须比上一次增加 > 0.3
```

### 预防检查清单

- [ ] 是否设定了迭代次数上限？
- [ ] 是否有趋势检测机制？
- [ ] 日志中是否记录了评分变化？
- [ ] 是否有突破不了某个评分时的降级方案？

## 风险4：知识库成为遗留负担

### 表现

- 知识库中堆积过时、错误的记录
- 反而误导系统给出错误推荐
- "垃圾进，垃圾出"效应
- 系统性能下降（检索变慢，推荐变差）

### 具体场景

```
知识库中的旧方案：
  问题：去水印
  方案：使用ImageMagick（已过时，新版本不兼容）
  评分：8.0（历史得分，但现在已不适用）

新用户请求：去水印
系统推荐：ImageMagick方案（因为评分最高）
结果：执行失败，用户失望
```

### 风险等级

🟡 **中** - 影响系统可信度和维护成本

### 应对策略

#### 1. 定期审计和清理知识库
- 定期（每月或每季度）审计知识库
- 检查评分是否还有效
- 删除或标记过时的方案

#### 2. 实现遗忘机制
长期不被使用的记录自动降低优先级：
```python
def update_knowledge_weight(solution, days_since_last_use):
    """
    长期不用的方案自动降权
    """
    if days_since_last_use > 180:  # 6个月未使用
        solution.weight *= 0.5
    if days_since_last_use > 365:  # 1年未使用
        solution.active = False
```

#### 3. 用户反馈机制来修正错误知识
```
用户使用推荐的方案失败
  ↓
系统询问：该方案对你有帮助吗？
  ↓
用户反馈："不行，不兼容"
  ↓
系统自动降低该方案的评分和推荐权重
```

#### 4. 版本控制和回滚
```
知识库的每个改动都有版本记录：
- 添加时间
- 修改时间
- 修改者
- 修改原因

如果发现某个版本的知识导致问题：
- 可以快速回滚到之前的版本
- 或标记为"不推荐"
```

#### 5. 知识库大小限制
- 不能无限增长
- 定期清理：保留有效期长的知识，删除低效知识
- 建议上限：初期100MB，后期可扩展到1GB

### 预防检查清单

- [ ] 知识库中是否有过期数据检测机制？
- [ ] 是否实现了遗忘机制（降权或删除）？
- [ ] 用户反馈是否能修正知识？
- [ ] 知识库的大小是否受到控制？

## 风险5：Agent间协作引入延迟

### 表现

- 为了协作导致每个请求的响应时间显著增加
- 原本5秒的请求变成了10秒
- 用户体验反而变差："为什么变慢了？"
- 系统并发能力下降

### 具体场景

```
原流程（无协作）：
  CodeGen: 2s → Execution: 1s → Evaluation: 2s = 5秒

新流程（有协作）：
  CodeGen: 2s → 快速评估: 1s → 决定是否执行: 0.5s 
    → Execution: 1s → 详细评估: 2s = 6.5秒

似乎只增加了1.5秒，但...

实际情况（协作变慢）：
  CodeGen: 2s → 等待Evaluation反馈: 2s （同步等待！）
    → Execution: 1s → 等待Evaluation反馈: 2s
    → 决定优化: 0.5s → CodeGen修改: 2s
  = 可能10秒+
```

### 风险等级

🟡 **中** - 影响系统性能和可用性

### 应对策略

#### 1. 严格控制协作的触发条件
- **不要让每个请求都协作**
- 只在高价值场景启用：
  - CodeGen完成后的快速评估（必须）
  - Evaluation质量差时反馈给CodeGen（可选）
  - Retrieval质量低时通知CodeGen（可选）

#### 2. 实现快速评估机制
区分两种评估，避免全量评估导致延迟：
```
轻量级评估（<1秒）：
  - 检查代码语法
  - 检查导入
  - 检查逻辑结构
  - 给出初步评分

vs

完整评估（执行完成后）：
  - 实际运行代码
  - 检查输出质量
  - 给出详细分析
```

#### 3. 异步协作机制
不要让Agent同步等待，改为异步：
```
CodeGen 完成代码
  ↓
发送"请评估"请求给Evaluation（不等待）
  ↓
同时准备执行
  ↓
收到Evaluation反馈后再根据反馈决定
```

#### 4. 协作优先级和超时
```
协作消息如果没有及时响应，不要一直等待：
  - 设定超时时间：2秒
  - 超时则使用默认行为（降级）
  - 记录日志，后续分析
```

#### 5. 性能监测和告警
```
监测指标：
- 单个请求的总耗时
- Agent间的通信时间
- 协作导致的额外延迟

告警阈值：
- 如果平均耗时 > 当前2倍 → 告警
- 如果协作引入 > 20% 延迟 → 告警
```

### 预防检查清单

- [ ] 是否限制了协作的触发条件？
- [ ] 是否实现了快速评估机制？
- [ ] 协作是否采用异步而不是同步？
- [ ] 是否有超时和降级机制？
- [ ] 性能是否被监测和告警？

## 风险总结与优先级

| # | 风险 | 等级 | 影响 | 预防难度 | 检测难度 |
|---|------|------|------|---------|---------|
| 1 | 复杂度爆炸 | 🔴高 | 可维护性 | 中等 | 低 |
| 2 | 决策错误 | 🟠中高 | 用户体验 | 中等 | 中等 |
| 3 | 迭代死循环 | 🟠中高 | 效率 | 低 | 低 |
| 4 | 知识库污染 | 🟡中 | 可信度 | 中等 | 高 |
| 5 | 协作延迟 | 🟡中 | 性能 | 中等 | 中等 |

## 风险应对的时间表

| 阶段 | 重点风险 | 应对措施 |
|------|---------|---------|
| 1（自动迭代） | 迭代死循环 | 设置上限、趋势检测 |
| 2（错误修复） | 复杂度增加 | 测试、日志、文档 |
| 3（决策规划） | 决策错误 | 保守策略、A/B测试 |
| 4（协作机制） | 协作延迟 | 异步、超时、监测 |
| 5（知识库） | 知识污染 | 审计、遗忘、反馈 |
